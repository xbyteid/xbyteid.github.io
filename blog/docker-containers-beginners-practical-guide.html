Docker for Developers: A Practical Guide to Containers in 2026

2026-06-24·15 min read·by xbyteid
DockerDevOpsContainersBackend

Docker containers are the standard unit of deployment in 2026. Whether you're deploying a Python API, a Next.js frontend, or a full microservices stack — understanding Docker is non-negotiable.

Dockerfile Fundamentals

A Dockerfile describes how to build your image. Here's a production Python API Dockerfile:

FROM python:3.12-slim AS base
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

Multi-Stage Builds

Multi-stage builds reduce final image size dramatically by separating build and runtime stages:

# Build stage
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Production stage
FROM node:20-alpine AS runner
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package.json .
EXPOSE 3000
CMD ["node", "dist/index.js"]

This can reduce a 1.2GB Node.js image to 200MB by excluding dev dependencies and source files from the final image.

Docker Compose

Compose orchestrates multiple containers for local development:

version: '3.8'
services:
  api:
    build: ./api
    ports:
      - "8000:8000"
    environment:
      DATABASE_URL: postgresql://user:pass@db:5432/mydb
      REDIS_URL: redis://redis:6379
    depends_on:
      db:
        condition: service_healthy
    volumes:
      - ./api:/app  # hot reload in dev

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user"]
      interval: 5s

  redis:
    image: redis:7-alpine

volumes:
  pgdata:

Volumes and Networking

Named volumes persist data across container restarts. Bind mounts map host directories into containers for development:

# Named volume (persistent, managed by Docker)
docker volume create mydata

# Bind mount (maps host path into container)
docker run -v $(pwd)/src:/app/src myimage

Docker creates a default bridge network for each compose project. Services reference each other by name: db, redis, api.

Production Best Practices

  • Use .dockerignore to exclude node_modules, .git, .env
  • Pin base image versions (python:3.12-slim, not latest)
  • Run as non-root user: RUN adduser --disabled-password appuser then USER appuser
  • Use COPY not ADD unless you need tar extraction
  • Order COPY instructions from least to most frequently changed (layer caching)
  • Scan images with docker scout or trivy for vulnerabilities
  • Use docker compose watch for automatic rebuilds during development
Pro tip: Use --mount type=cache for pip/npm caches in multi-stage builds. This persists the package cache across builds, making rebuilds 5-10x faster.

Docker simplifies deployment to a single command: docker compose up -d. Combined with GitHub Actions for CI/CD, you get a fully automated pipeline from push to production.
