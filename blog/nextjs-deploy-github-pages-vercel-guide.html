Deploy Next.js Apps: GitHub Pages vs Vercel vs Self-Hosted in 2026

2026-06-24·11 min read·by xbyteid
Next.jsDeploymentDevOpsFrontend

Next.js offers multiple deployment strategies, each with different trade-offs. This guide compares the three main options and helps you pick the right one for your project.

Option 1: Static Export to GitHub Pages

For fully static sites (no SSR/ISR), export to static HTML and host free on GitHub Pages:

// next.config.js
/** @type {import('next').NextConfig} */
const nextConfig = {
  output: 'export',
  images: { unoptimized: true },
  basePath: '/my-repo',  // for GitHub Pages subpath
}
module.exports = nextConfig

Deploy with GitHub Actions:

# .github/workflows/deploy.yml
name: Deploy to GitHub Pages
on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci && npm run build
      - uses: actions/upload-pages-artifact@v3
        with: { path: './out' }

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4

Pros: Free hosting, simple setup, great for portfolios/blogs. Cons: No SSR, no API routes, no dynamic content.

Option 2: Vercel

Vercel is the platform built by the Next.js team. Zero-config deployment with full feature support:

# Install Vercel CLI
npm i -g vercel

# Deploy to preview
vercel

# Deploy to production
vercel --prod

Vercel automatically detects Next.js and configures everything: ISR, SSR, API routes, middleware, edge functions, image optimization.

Pros: Zero config, full Next.js features, edge network, preview deployments per PR. Cons: Free tier limits (100GB bandwidth), can get expensive at scale.

Option 3: Self-Hosted with Docker

For full control, deploy Next.js in a Docker container behind nginx or Caddy:

FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public
EXPOSE 3000
CMD ["node", "server.js"]

Enable standalone output in next.config.js:

const nextConfig = {
  output: 'standalone',
}

Pros: Full control, no vendor lock-in, cost-effective at scale. Cons: You manage SSL, scaling, CDN, and updates yourself.

Comparison Table

  • GitHub Pages: Static only, free, simplest setup, no server costs
  • Vercel: Full features, free tier, fastest deploy, vendor lock-in risk
  • Self-Hosted: Full control, any feature, requires DevOps knowledge
When to choose what: Portfolio/blog → GitHub Pages. Startup/SaaS → Vercel for speed, migrate to self-hosted when costs justify it. Enterprise → Self-hosted for compliance and control.

For most developers, start with Vercel for convenience. If bandwidth costs become an issue or you need data sovereignty, Docker + a VPS gives you unlimited flexibility at a predictable $5-20/month.
