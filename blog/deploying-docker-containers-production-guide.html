Deploying Docker Containers to Production: A Complete Guide

2026-07-01 · 15 min read · by xbyteid
Node.jsExpressAPIBackend

Node.js continues to be one of the most popular choices for building APIs in 2026. With the latest Express 5 release and Node 22+ features like native ESM and the built-in test runner, building REST APIs has never been more productive. This tutorial walks through building a production-grade API from scratch.

Project Setup

Start with a clean project structure. We'll use ES modules and TypeScript throughout:

mkdir api-server && cd api-server
npm init -y
npm install express cors helmet morgan
npm install -D typescript @types/express @types/node tsx nodemon

Update package.json to use ESM:

{
  "name": "api-server",
  "type": "module",
  "scripts": {
    "dev": "nodemon --exec tsx src/index.ts",
    "build": "tsc",
    "start": "node dist/index.js"
  }
}

Application Architecture

Organize your code by feature, not by type. This scales better as your API grows:

src/
├── index.ts              # Entry point
├── app.ts                # Express app setup
├── config/
│   └── env.ts            # Environment variables
├── middleware/
│   ├── errorHandler.ts   # Global error handler
│   ├── validate.ts       # Request validation
│   └── auth.ts           # Authentication
├── modules/
│   ├── users/
│   │   ├── users.routes.ts
│   │   ├── users.controller.ts
│   │   ├── users.service.ts
│   │   └── users.validation.ts
│   └── posts/
│       ├── posts.routes.ts
│       ├── posts.controller.ts
│       ├── posts.service.ts
│       └── posts.validation.ts
└── utils/
    ├── ApiError.ts
    └── asyncHandler.ts

Each module has its own routes, controller (HTTP layer), service (business logic), and validation schemas. This separation makes testing straightforward and keeps components focused.

Setting Up Express

Create the Express app with essential middleware:

// src/app.ts
import express from 'express';
import cors from 'cors';
import helmet from 'helmet';
import morgan from 'morgan';
import { errorHandler } from './middleware/errorHandler.js';
import { usersRouter } from './modules/users/users.routes.js';
import { postsRouter } from './modules/posts/posts.routes.js';

const app = express();

// Security headers
app.use(helmet());

// CORS - restrict in production
app.use(cors({
  origin: process.env.CORS_ORIGIN || '*',
  credentials: true
}));

// Logging
app.use(morgan(process.env.NODE_ENV === 'production' ? 'combined' : 'dev'));

// Body parsing
app.use(express.json({ limit: '10mb' }));

// Health check
app.get('/health', (req, res) => {
  res.json({ status: 'ok', timestamp: new Date().toISOString() });
});

// Routes
app.use('/api/v1/users', usersRouter);
app.use('/api/v1/posts', postsRouter);

// 404 handler
app.use((req, res) => {
  res.status(404).json({ error: 'Not found' });
});

// Global error handler (must be last)
app.use(errorHandler);

export { app };

The app.ts file exports the app without starting the server. This lets you import it for testing without binding to a port.

Async Error Handling

Express 4 doesn't catch async errors automatically. Use a simple wrapper:

// src/utils/asyncHandler.ts
import { Request, Response, NextFunction } from 'express';

export const asyncHandler = (
  fn: (req: Request, res: Response, next: NextFunction) => Promise<any>
) => (req: Request, res: Response, next: NextFunction) => {
  fn(req, res, next).catch(next);
};

Wrap every async route handler with asyncHandler and any thrown error will reach your global error handler. Express 5 handles this natively, but if you're still on Express 4, this wrapper is essential.

Request Validation with Zod

Zod is the standard for request validation in the Node.js ecosystem. Define schemas once and derive TypeScript types automatically:

// src/modules/users/users.validation.ts
import { z } from 'zod';

export const createUserSchema = z.object({
  body: z.object({
    email: z.string().email('Invalid email address'),
    name: z.string().min(2).max(100),
    role: z.enum(['user', 'admin']).default('user')
  })
});

export const updateUserSchema = z.object({
  params: z.object({ id: z.string().uuid() }),
  body: z.object({
    email: z.string().email().optional(),
    name: z.string().min(2).max(100).optional()
  }).refine(data => Object.keys(data).length > 0, {
    message: 'At least one field must be provided'
  })
});

export type CreateUserInput = z.infer<typeof createUserSchema>['body'];
export type UpdateUserInput = z.infer<typeof updateUserSchema>['body'];

Create a validation middleware that parses and validates requests:

// src/middleware/validate.ts
import { Request, Response, NextFunction } from 'express';
import { AnyZodObject, ZodError } from 'zod';

export const validate = (schema: AnyZodObject) => (
  req: Request, res: Response, next: NextFunction
) => {
  try {
    schema.parse({
      body: req.body,
      query: req.query,
      params: req.params
    });
    next();
  } catch (error) {
    if (error instanceof ZodError) {
      return res.status(400).json({
        error: 'Validation failed',
        details: error.errors.map(e => ({
          field: e.path.join('.'),
          message: e.message
        }))
      });
    }
    next(error);
  }
};

Building the Controller and Service

Keep controllers thin — they parse input and send responses. Services contain the business logic:

// src/modules/users/users.service.ts
import { CreateUserInput } from './users.validation.js';

// ponytail: in-memory store, swap for database (Drizzle/Prisma) in production
const users: Map<string, any> = new Map();

export const usersService = {
  create(data: CreateUserInput) {
    const id = crypto.randomUUID();
    const user = { id, ...data, createdAt: new Date().toISOString() };
    users.set(id, user);
    return user;
  },

  findAll(page = 1, limit = 20) {
    const all = Array.from(users.values());
    const start = (page - 1) * limit;
    return {
      data: all.slice(start, start + limit),
      total: all.length,
      page,
      totalPages: Math.ceil(all.length / limit)
    };
  },

  findById(id: string) {
    return users.get(id) || null;
  },

  delete(id: string) {
    return users.delete(id);
  }
};
// src/modules/users/users.controller.ts
import { Response } from 'express';
import { asyncHandler } from '../../utils/asyncHandler.js';
import { usersService } from './users.service.js';

export const usersController = {
  create: asyncHandler(async (req, res) => {
    const user = usersService.create(req.body);
    res.status(201).json(user);
  }),

  findAll: asyncHandler(async (req, res) => {
    const page = parseInt(req.query.page as string) || 1;
    const limit = parseInt(req.query.limit as string) || 20;
    const result = usersService.findAll(page, limit);
    res.json(result);
  }),

  findById: asyncHandler(async (req, res) => {
    const user = usersService.findById(req.params.id);
    if (!user) return res.status(404).json({ error: 'User not found' });
    res.json(user);
  }),

  delete: asyncHandler(async (req, res) => {
    const deleted = usersService.delete(req.params.id);
    if (!deleted) return res.status(404).json({ error: 'User not found' });
    res.status(204).send();
  })
};

Route Definitions

Wire everything together with routes:

// src/modules/users/users.routes.ts
import { Router } from 'express';
import { usersController } from './users.controller.js';
import { validate } from '../../middleware/validate.js';
import { createUserSchema } from './users.validation.js';

const router = Router();

router.post('/', validate(createUserSchema), usersController.create);
router.get('/', usersController.findAll);
router.get('/:id', usersController.findById);
router.delete('/:id', usersController.delete);

export { router as usersRouter };

Global Error Handler

A centralized error handler catches all errors and returns consistent responses:

// src/middleware/errorHandler.ts
import { Request, Response, NextFunction } from 'express';

export class ApiError extends Error {
  constructor(public statusCode: number, message: string) {
    super(message);
    this.name = 'ApiError';
  }
}

export const errorHandler = (
  err: Error,
  req: Request,
  res: Response,
  next: NextFunction
) => {
  console.error(`[ERROR] ${req.method} ${req.path}:`, err.message);

  if (err instanceof ApiError) {
    return res.status(err.statusCode).json({ error: err.message });
  }

  // Don't leak internal details in production
  const message = process.env.NODE_ENV === 'production'
    ? 'Internal server error'
    : err.message;

  res.status(500).json({ error: message });
};

Testing with the Built-in Test Runner

Node 22+ has a built-in test runner — no need for Jest or Mocha for basic tests:

// src/__tests__/users.test.ts
import { describe, it, before, after } from 'node:test';
import assert from 'node:assert/strict';

const BASE = 'http://localhost:3000/api/v1';

describe('Users API', () => {
  it('creates a user', async () => {
    const res = await fetch(`${BASE}/users`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        email: 'test@example.com',
        name: 'Test User'
      })
    });
    assert.equal(res.status, 201);
    const data = await res.json();
    assert.ok(data.id);
    assert.equal(data.email, 'test@example.com');
  });

  it('returns 400 for invalid input', async () => {
    const res = await fetch(`${BASE}/users`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email: 'not-an-email' })
    });
    assert.equal(res.status, 400);
  });

  it('lists users with pagination', async () => {
    const res = await fetch(`${BASE}/users?page=1&limit=10`);
    assert.equal(res.status, 200);
    const data = await res.json();
    assert.ok(Array.isArray(data.data));
    assert.ok(typeof data.total === 'number');
  });
});

Run with node --test src/__tests__/**/*.test.ts — zero dependencies needed.

Environment Configuration

Use dotenv or Node 22's built-in --env-file flag:

// src/config/env.ts
const env = {
  PORT: parseInt(process.env.PORT || '3000'),
  NODE_ENV: process.env.NODE_ENV || 'development',
  CORS_ORIGIN: process.env.CORS_ORIGIN || '*',
  DATABASE_URL: process.env.DATABASE_URL || ''
};

export default env;

Run with node --env-file=.env dist/index.js — no dotenv package needed on Node 20+.

Production Checklist

  • Security headers — helmet() covers most (CSP, HSTS, X-Frame-Options)
  • Rate limiting — use express-rate-limit to prevent abuse
  • Request logging — morgan for HTTP, structured JSON logs for app-level logging
  • Graceful shutdown — handle SIGTERM, drain connections before exit
  • Process management — use PM2 or Docker restart policies, not just node
  • Health endpoint — return 200 when app can serve traffic, include dependency checks
  • CORS — set explicit origins in production, never use *
  • Input size limits — express.json({ limit: '10mb' }) prevents memory exhaustion
Pro tip: Express 5 (released late 2025) finally catches async errors natively. If you're starting a new project in 2026, use Express 5 — you can drop the asyncHandler wrapper entirely.

Next Steps

This gives you a solid foundation. From here, add a real database (Drizzle ORM with PostgreSQL is the current sweet spot), authentication with JWT or session tokens, and CI/CD with GitHub Actions. The architecture scales — swap the in-memory store for a database, add a Redis cache layer, and deploy behind an Nginx reverse proxy when you're ready for production traffic.

Docker has revolutionized how we deploy applications. In this comprehensive guide, we'll cover everything you need to know about deploying Docker containers to production environments.

Why Docker for Production?

Docker containers provide consistency across environments, easy scaling, and efficient resource utilization. Here's why you should consider Docker for production:

  • Consistency: Same environment in development and production
  • Isolation: Applications run in isolated containers
  • Scalability: Easy horizontal scaling with orchestration
  • Efficiency: Lower overhead than virtual machines

Building Optimized Docker Images

The first step to production deployment is building efficient Docker images:

# Use multi-stage builds
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/node_modules ./node_modules
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

Best Practices for Image Building

  • Use specific base image tags (not latest)
  • Minimize layers by combining RUN commands
  • Use multi-stage builds to reduce final image size
  • Include only necessary files with .dockerignore

Docker Compose for Production

Docker Compose simplifies multi-container deployments:

version: '3.8'
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb
    depends_on:
      - db
      - redis
    restart: unless-stopped

  db:
    image: postgres:15-alpine
    volumes:
      - postgres_data:/var/lib/postgresql/data
    environment:
      - POSTGRES_PASSWORD=secure_password
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    restart: unless-stopped

volumes:
  postgres_data:

Health Checks and Monitoring

Implement proper health checks for production containers:

HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3   CMD curl -f http://localhost:3000/health || exit 1

Security Considerations

Production Docker deployments require security best practices:

  • Run containers as non-root users
  • Scan images for vulnerabilities
  • Use secrets management for sensitive data
  • Limit container resources with --memory and --cpus

Deployment Strategies

Choose the right deployment strategy for your needs:

  • Rolling Updates: Gradually replace old containers with new ones
  • Blue-Green: Switch between two identical environments
  • Canary: Deploy to a subset of users first

Conclusion

Docker provides a robust platform for production deployments. By following these best practices, you can ensure reliable, scalable, and secure container deployments.

  • Health endpoint — return 200 when app can serve traffic, include dependency checks
  • CORS — set explicit origins in production, never use *
  • Input size limits — express.json({ limit: '10mb' }) prevents memory exhaustion
Pro tip: Express 5 (released late 2025) finally catches async errors natively. If you're starting a new project in 2026, use Express 5 — you can drop the asyncHandler wrapper entirely.

Next Steps

This gives you a solid foundation. From here, add a real database (Drizzle ORM with PostgreSQL is the current sweet spot), authentication with JWT or session tokens, and CI/CD with GitHub Actions. The architecture scales — swap the in-memory store for a database, add a Redis cache layer, and deploy behind an Nginx reverse proxy when you're ready for production traffic.
