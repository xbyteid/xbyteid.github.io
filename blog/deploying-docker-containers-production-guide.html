Deploying Docker Containers to Production: A Complete Guide

2026-07-02 · 16 min read · by xbyteid
DockerDevOpsContainersProduction

Running docker run on your laptop is easy. Deploying to production reliably — with health checks, proper logging, zero-downtime deploys, and security hardening — is a different challenge entirely. This guide covers everything you need to take Docker containers from development to production.

Multi-Stage Builds

A multi-stage build separates build dependencies from the runtime image. This cuts image sizes by 60-90% and removes attack surface:

# ---- Build stage ----
FROM node:22-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --ignore-scripts
COPY . .
RUN npm run build
RUN npm prune --production

# ---- Production stage ----
FROM node:22-alpine AS production
WORKDIR /app

# Don't run as root
RUN addgroup -g 1001 -S appgroup && \
    adduser -S appuser -u 1001

COPY --from=builder --chown=appuser:appgroup /app/dist ./dist
COPY --from=builder --chown=appuser:appgroup /app/node_modules ./node_modules
COPY --from=builder --chown=appuser:appgroup /app/package.json ./

USER appuser

EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

CMD ["node", "dist/index.js"]

The final image has only Node.js runtime, production dependencies, and your compiled code. No source maps, dev dependencies, or build tools.

Optimizing Layer Caching

Docker caches layers. Order your COPY instructions so the least-changing layers come first:

  1. COPY package*.json — changes only when dependencies change
  2. RUN npm ci — cached if package files unchanged
  3. COPY . . — changes on every code update
  4. RUN npm run build — rebuilds only when code changes

This means npm ci only re-runs when you actually change dependencies, not on every commit.

Docker Compose for Production

Docker Compose works for single-server deployments. Use a dedicated production override:

# docker-compose.prod.yml
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
    restart: unless-stopped
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - DATABASE_URL=postgresql://user:pass@db:5432/app
      - REDIS_URL=redis://cache:6379
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: '0.5'
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

  db:
    image: postgres:16-alpine
    restart: unless-stopped
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: app
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user -d app"]
      interval: 10s
      timeout: 5s
      retries: 5

  cache:
    image: redis:7-alpine
    restart: unless-stopped
    volumes:
      - redisdata:/data
    command: redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru

volumes:
  pgdata:
  redisdata:

Deploy with docker compose -f docker-compose.prod.yml up -d. The depends_on with health checks ensures your app won't start until the database is accepting connections.

Health Checks

Health checks let Docker (and orchestrators) know if a container can serve traffic. Without them, a container running a crashed app looks "healthy":

# In Dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

# For Node.js apps, add a /health endpoint:
app.get('/health', async (req, res) => {
  // Check critical dependencies
  const dbOk = await checkDatabase();
  const cacheOk = await checkRedis();

  const status = dbOk && cacheOk ? 200 : 503;
  res.status(status).json({
    status: status === 200 ? 'ok' : 'degraded',
    db: dbOk,
    cache: cacheOk,
    uptime: process.uptime()
  });
});
Important: Health check endpoint must return quickly (< 3s). Don't run expensive queries. Check basic connectivity, not full business logic.

Logging Strategy

Write logs to stdout/stderr — never to files inside the container. Docker captures stdout and forwards it to your logging driver:

# Logging configuration in compose
services:
  app:
    logging:
      driver: json-file
      options:
        max-size: "10m"    # Rotate at 10MB
        max-file: "5"      # Keep 5 rotated files
        tag: "{{.Name}}"

For centralized logging, use the syslog or fluentd driver to ship logs to an external system like Loki, ELK, or Datadog. Your app code doesn't change — just the Docker logging configuration.

Structured Logging in Node.js

Use JSON log format so log aggregators can parse it:

// Using pino for structured logging
import pino from 'pino';

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  formatters: {
    level(label) {
      return { level: label };
    }
  },
  timestamp: pino.stdTimeFunctions.isoTime
});

logger.info({ userId: '123', action: 'login' }, 'User logged in');
// Output: {"level":"info","time":"2026-07-02T10:30:00.000Z","userId":"123","action":"login","msg":"User logged in"}

Security Hardening

Default Docker settings are permissive. Tighten them for production:

Non-Root User

# Create and switch to non-root user
RUN addgroup -g 1001 -S appgroup && \
    adduser -S appuser -u 1001 -G appgroup
USER appuser

Always run as non-root. If your app doesn't need root privileges (and it almost never does), don't give it root.

Read-Only Filesystem

# In docker-compose.prod.yml
services:
  app:
    read_only: true
    tmpfs:
      - /tmp
    security_opt:
      - no-new-privileges:true

A read-only filesystem prevents an attacker (or a bug) from writing to unexpected paths. The tmpfs mount gives the app a writable /tmp if needed.

Image Scanning

Scan your images for known vulnerabilities before deployment:

# Using Docker Scout (built into Docker Desktop)
docker scout cves myapp:latest

# Using Trivy
docker run --rm -v /var/run/docker.sock:/var/run/docker.sock \
  aquasec/trivy image myapp:latest

# Scan in CI pipeline
docker scout cves --only-severity critical,high myapp:latest

Fix critical and high severity issues before deploying. Low and informational findings can be triaged.

Zero-Downtime Deployments

For single-server setups, use the blue-green approach with Docker Compose:

#!/bin/bash
# deploy.sh — zero-downtime deployment script

set -e

IMAGE="myapp:latest"
SERVICE="app"

# Build new image
docker compose -f docker-compose.prod.yml build $SERVICE

# Start new container alongside old one
docker compose -f docker-compose.prod.yml up -d --no-deps --scale ${SERVICE}=2 $SERVICE

# Wait for new container to be healthy
echo "Waiting for new container to pass health check..."
sleep 10

# Stop old container (oldest instance)
OLD=$(docker compose -f docker-compose.prod.yml ps -q $SERVICE | head -1)
docker stop $OLD
docker rm $OLD

echo "Deployment complete"

For multi-server deployments, you'll want proper orchestration. Docker Swarm is built into Docker and handles rolling updates natively:

# Initialize swarm (once)
docker swarm init

# Deploy stack
docker stack deploy -c docker-compose.prod.yml myapp

# Update with rolling restart
docker service update --image myapp:v2.1 --update-parallelism 1 --update-delay 10s myapp_app

Resource Limits

Always set memory and CPU limits. Without them, a single container can starve the host:

deploy:
  resources:
    limits:
      memory: 512M
      cpus: '0.5'
    reservations:
      memory: 256M
      cpus: '0.25'

Start with conservative limits and increase based on observed usage. Monitor with docker stats or a monitoring stack like Prometheus + Grafana.

Volume Management

Use named volumes for persistent data, never bind mounts in production:

volumes:
  pgdata:
    driver: local
  redisdata:
    driver: local

Named volumes are portable, managed by Docker, and survive container removal. Back them up with docker run --rm -v pgdata:/data -v $(pwd):/backup alpine tar czf /backup/pgdata-backup.tar.gz /data.

Networking

Use Docker networks to isolate services. Only expose ports that need external access:

services:
  app:
    ports:
      - "80:3000"   # Only app needs external access
    networks:
      - frontend
      - backend

  db:
    # No ports exposed to host!
    networks:
      - backend

  cache:
    # No ports exposed to host!
    networks:
      - backend

networks:
  frontend:
  backend:
    internal: true   # No external access

The internal: true flag on the backend network means containers on that network can't reach the internet — only each other.

Monitoring in Production

At minimum, track these metrics:

  • Container health — health check pass/fail rate
  • Resource usage — CPU, memory, disk per container
  • Restart count — frequent restarts indicate a bug
  • Response times — p50, p95, p99 latency from inside the container
  • Error rate — 4xx/5xx responses per minute

Docker's built-in docker stats gives quick checks. For production, use cAdvisor + Prometheus + Grafana for dashboards and alerts.

Checklist before go-live: Multi-stage build ✓ Non-root user ✓ Health check ✓ Resource limits ✓ Log rotation ✓ Read-only filesystem ✓ Image scan ✓ Network isolation ✓ Restart policy ✓

Deploying Docker to production is about eliminating surprises. Multi-stage builds keep images small. Health checks catch failures early. Resource limits prevent runaway containers. Security hardening reduces attack surface. Get these fundamentals right, and Docker becomes a reliable deployment platform — not a source of 3 AM pages.
