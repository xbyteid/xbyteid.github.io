Solana Token Extensions (Token-2022): Security Deep Dive

2026-06-24 · 16 min read · by xbyteid
Solana Token-2022 Smart Contracts Security

Solana's Token-2022 program (also called Token Extensions) introduced powerful new primitives: transfer hooks, confidential transfers, metadata pointers, and more. These features unlocked sophisticated DeFi and gaming use cases — but they also introduced entirely new attack surfaces.

Token-2022 Quick Refresher

Token-2022 lives at a different program ID than the original SPL Token program. It is backward compatible in many ways but adds optional "extensions" stored in the mint and token account data.

Key extensions relevant to security:

  • Transfer Hook — execute custom program logic on every transfer
  • Non-Transferable — soulbound style tokens
  • Metadata Pointer + Token Metadata
  • Confidential Transfers (using ElGamal)
  • Interest Bearing / Scaled UI Amount
  • Mint Close Authority, Permanent Delegate, etc.

High-Impact Vulnerability Classes in 2025–2026

1. Callback Reentrancy via Transfer Hooks

The transfer hook is the most dangerous new primitive. On every transfer (and sometimes mint/burn), the token program CPI-calls your hook program.

The classic pattern:

// Attacker controlled account
transfer(from=attackerAta, to=victimAta, amount=1)
  → token2022 CPI
    → hook_program.on_transfer(...) 
       // inside hook attacker can now call back into protocol
       // before balances are finalized
Real impact: Many early protocols that added transfer hooks for taxes, royalties, or access control did not properly handle reentrant calls. The result was double-spend of voting power, double-claim of rewards, and broken accounting.

Mitigation: Use a reentrancy guard (simple boolean or PDA lock), avoid external calls in hooks if possible, or use transfer_checked patterns with explicit post-hook accounting reconciliation.

2. Authority Confusion & Permanent Delegate Abuse

Token-2022 added Permanent Delegate and Close Mint Authority. When these are set incorrectly, entire supplies can be moved or the mint can be closed after distribution, creating permanent rug vectors.

Common mistake: teams set Permanent Delegate to a "multisig" that is actually a single key controlled by one founder. Or the delegate key is never rotated after launch.

3. Metadata Pointer Hijacking

Because metadata can now live in a separate account pointed to by the mint, an attacker who can update the pointer (if authority is not revoked) can change name/symbol or even point to malicious on-chain metadata containing phishing links or fake instructions.

4. Confidential Transfer Privacy vs Auditability Tradeoffs

Confidential transfers hide amounts. This is great for privacy but terrible if your protocol relies on being able to read balances on-chain for liquidation logic, reward distribution, or governance power. Many protocols added silent failures or defaulted to public amounts incorrectly.

Practical Audit Checklist for Token-2022 Protocols

  • Is every extension authority revoked or properly timelocked?
  • Does any transfer hook perform external CPI? Is it reentrancy protected?
  • Are hook programs immutable and verified on-chain?
  • Does the protocol correctly handle both legacy SPL and Token-2022 accounts?
  • Are scaled UI amounts and interest bearing extensions handled in all accounting math?
  • Is the metadata pointer immutable after launch?
  • Can an attacker create a malicious mint with the same symbol that passes superficial UI checks?

Example: Simple Safe Transfer Hook Skeleton (Anchor)

#[interface]
pub trait TransferHook {
    fn on_transfer(&self, amount: u64) -> Result<()>;
}

#[program]
pub mod safe_hook {
    use super::*;
    pub fn on_transfer(ctx: Context<OnTransfer>, amount: u64) -> Result<()> {
        // Reentrancy guard + state validation
        let state = &mut ctx.accounts.state;
        require!(!state.in_hook, ErrorCode::Reentrancy);
        state.in_hook = true;
        
        // business logic here...
        
        state.in_hook = false;
        Ok(())
    }
}

Common Exploit Techniques Observed in the Wild

  1. Flash-loan style callback abuse: borrow huge amount, trigger hook, manipulate internal accounting inside the hook window, repay.
  2. Delegate sandwich: sandwich legitimate user transfer with delegate operations when Permanent Delegate exists.
  3. Metadata upgrade grief: point metadata to a program that returns huge strings causing UI crashes for indexers.
  4. Account size / extension desync: create accounts with partial extensions initialized, causing downstream deserialization panics.
2026 recommendation: Treat every Token-2022 mint with hooks as high risk. Require formal verification or extensive fuzzing on the hook program before mainnet use. Prefer immutable programs.

Conclusion

Token Extensions are a massive leap forward for Solana, but they move a lot of security responsibility from the runtime into user programs. The same way early ERC-777 hooks created chaos on Ethereum, transfer hooks are the new battleground.

Build defensively. Revoke authorities. Guard against reentrancy. And remember: the most dangerous code is the code that only runs on every single transfer.
