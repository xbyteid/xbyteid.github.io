GitHub Actions CI/CD: Automate Your Dev Workflow from Zero

2026-06-24·12 min read·by xbyteid
CI/CDGitHubAutomationDevOps

GitHub Actions lets you automate everything from linting to deployment directly from your repository. Here's how to build real CI/CD pipelines that actually work in production.

Workflow Basics

Every workflow lives in .github/workflows/ as a YAML file. Here's a minimal Node.js CI pipeline:

name: CI
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm test
      - run: npm run lint

Secrets Management

Never hardcode credentials. Use GitHub Secrets for tokens, API keys, and deploy credentials:

jobs:
  deploy:
    runs-on: ubuntu-latest
    needs: test
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - name: Deploy
        env:
          DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
          API_KEY: ${{ secrets.API_KEY }}
        run: ./deploy.sh

Access secrets via ${{ secrets.MY_SECRET }}. They're masked in logs automatically. Use Environment secrets for per-stage (staging/production) separation.

Matrix Builds

Test across multiple versions and OS in parallel:

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        node-version: [18, 20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci && npm test

Caching Dependencies

Cache node_modules, pip packages, or any directory to speed up workflows:

    steps:
      - uses: actions/cache@v4
        with:
          path: ~/.npm
          key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
          restore-keys: ${{ runner.os }}-node-

Or use the built-in cache option in actions/setup-node and actions/setup-python — simpler and handles key generation automatically.

Deployment Workflows

Deploy to different environments based on branch or tag:

name: Deploy
on:
  push:
    tags: ['v*']

jobs:
  deploy-prod:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: npm run build
      - name: Deploy to Vercel
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          vercel-args: '--prod'

Reusable Workflows

Avoid duplication with reusable workflows. Create .github/workflows/reusable-deploy.yml:

on:
  workflow_call:
    inputs:
      environment:
        required: true
        type: string

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ inputs.environment }}
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh ${{ inputs.environment }}

Call it from other workflows: uses: ./.github/workflows/reusable-deploy.yml

Pro tip: Use concurrency groups to cancel in-progress runs when you push again: concurrency: { group: deploy-${{ github.ref }}, cancel-in-progress: true }

GitHub Actions is the most integrated CI/CD platform for GitHub-hosted projects. With caching, matrix builds, and reusable workflows, you can build pipelines that rival any dedicated CI service — without leaving your repo.
